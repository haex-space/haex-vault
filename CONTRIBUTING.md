# Contributing

## Branching

We use a develop-first flow:

```
feature/fix branch ──PR──▶ develop ──release-please──▶ main ──tag──▶ release
                                                         │
                                                         ▼
                                                   sync-develop PR
                                                         │
                                                         ▼
                                                      develop
```

### Where to send your PR

| Your branch              | Base | Notes                                                                |
|--------------------------|------|----------------------------------------------------------------------|
| `feat/*`, `fix/*`, etc.  | `develop` | Default. Everything that isn't a release or hotfix.             |
| `release-please--*`      | `main`    | Auto-generated by `release-please.yml`. Don't create by hand.   |
| `sync-*`                 | `main` or `develop` | Auto-generated sync helpers (e.g. `sync-develop/v2.2.0`). |
| `hotfix/*`               | `main`    | Production-critical fixes that can't wait for the next release. Open a second PR to also bring the same fix into `develop`. |

PRs targeting `main` from any other head branch are rejected by the
`Enforce develop-first` workflow (`.github/workflows/enforce-main-base.yml`).

### Creating a PR

```bash
# Default base is `develop`.
gh pr create --base develop --title "fix(foo): bar"

# If you forgot and targeted main by accident:
gh pr edit <number> --base develop
```

### Release flow (maintainers)

1. `release-please.yml` runs on every push to `main`, opens or updates a
   "Release vX.Y.Z" PR with `release-please--branches--main` as head.
2. Merge that PR — release-please tags the merge commit, builds artifacts,
   and automatically opens a `sync-develop/vX.Y.Z` PR carrying the version
   bumps back into `develop`.
3. Merge the sync PR. `develop` is now caught up with `main`.

The release-please PR is the **only** way new features land on `main`
(via the develop → release-please flow).

## Commits

- Conventional Commits (`feat:`, `fix:`, `chore:`, `refactor:`, `test:`, `docs:`, `ci:`).
- One bug per commit when feasible — release-please reads commit messages
  to generate the changelog.
- Commit messages in English, short and specific.

## Code review

- All PRs need at least one approving review or CodeRabbit pass before merging.
- CodeRabbit auto-reviews PRs targeting `develop` (it skips non-default-base PRs).
